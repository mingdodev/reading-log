# 3. 프로세스 관리

## 프로세스

> 실행 중인 프로그램. 사용 중인 자원과 현재 상태를 전부 포함하는 개념이다.

리눅스에서는 프로세스와 스레드를 구분하지 않는다. 스레드는 그냥 자원을 공유하는 프로세스로 취급된다. 전부 task의 개념

아래는 스레드가 공유하지 않는 자원들. 생각해보면 당연하다!

- 프로세스 레지스터
  - CPU 내부의 임시 저장소
  - 프로그램 카운터
  - 각 스레드가 독립적으로 가진다. (고유한 가상 프로세서 할당)

- 프로세스 스택
  - 메모리 영역의 일부
  - 함수 호출 기록
  - 스레드는 메모리 중 코드, 데이터, 힙 영역은 공유하지만, 스택은 각 스레드가 독립적으로 가진다. (각자의 함수 호출 흐름 관리)

### 태스크가 CPU 점유에서 내려올 때, 점유했던 메모리가 해제될 때도 있나?

컨텍스트 스위칭: CPU 레지스터에 있는 값들을 커널 스택의 task_struct(PCB/TCB)에 복사해 저장하고, 다음 태스크의 task_struct 정보를 가져와 레지스터에 덮어씌우고 실행. 메모리는 그대로 유지

그러나 메모리가 극도로 부족하면, 오랫동안 실행되지 않은 프로세스의 메모리 내용을 하드디스크의 스왑 영역으로 보낼 수 있음

- **Swap 영역**
    - 디스크의 일부분을 부족한 물리 메모리 대신 임시 저장 공간으로 활용 (Swap-in / Swap-out)
    - 메모리가 부족하면 프로세스가 더 이상 연산을 진행할 수 없음 → 전체 시스템이 응답 불가 상태가 될 수 있기에 마련해 둔 공간
    - 당연히 메모리보다 느리니 시스템 성능 저하가 있다.
    - RAM의 여유 공간이 커널이 정한 Low 워터마크(임계치) 아래로 떨어질 때 시작
        - `kswapd` : 커널 내부의 백그라운드 스왑 데몬이 활성화되어 작업을 진행.
- **OOM Killer**
    - 스왑 영역도 꽉 차거나, 메모리 확보가 감당 불가능할 때(Min 워터마크 아래로) 특정 프로세스를 강제로 종료하고 메모리를 완전히 회수
    - oom_score 점수를 매겨, 시스템 운영에 덜 중요하거나 메모리를 가장 많이 잡아먹는 프로세스를 kill
        
        > cf. 점수를 매기는 여러 함수가 있는데, 스왑 영역을 얼마나 차지하는가(get_mm_counter)도 고려한다. 그만큼 그 프로세스가 자원을 많이 쓰고 있다는 뜻이기도 하고, 많이 쓰는 애를 종료시키면 자원 확보도 많이 될 것
        > 


시스템의 메모리 부족은 어떻게 대응해야 할까?

- 어떤 프로세스가 메모리를 과도하게 사용하고 있는지 확인
→ `top, htop, ps, pmap`
- 주기적으로 스왑 영역 사용이 일어난다면 메모리 증설
→ 적어도 Swap Used 이상의 RAM 증설 필요

### 커널 영역 데이터는 디스크냐 메모리냐?

메모리(RAM).

- 커널도 프로그램이다. 부팅 시 디스크에 있던 커널 파일이 메모리의 특정 구역(커널 영역)에 상주하게 됨
- 따라서 `task_struct` 도 RAM의 커널 전용 공간에 기록되고 읽힌다. 스케쥴링은 1초에도 수백 번씩 일어나는데 느린 디스크로는 감당 불가

### 메모리 구조

- 커널 영역
- 사용자 영역
  - 스택영역, 힙영역, 데이터영역, 코드영역
  - 프로세스마다 각 영역이 존재함

**언제 영역이 할당되나요?** 컴퓨터가 부팅될 때 커널이 Page Table을 구성하면서 가상 메모리 경계를 설정함

### 구체적으로 어떻게? 개발자가 설정을 통해 가상 메모리 경계를 변경할 수 있나?

- 컴퓨터가 부팅될 때 커널이 전체 가상 메모리 공간을 나누는 Page Table을 구성하면서 사용자/커널 공간 경계를 나눔
- 가상 메모리 경계 설정 (VMSplit): 보통 기본 비율이 있음. 보통 3:1(커널)

개발자가 설정을 통해 가상 메모리 경계를 변경할 수 있나? 네.

- menuconfiig의 memory split 항목에서 변경할 수 있음

## 프로세스 생성 과정

부팅 직후는 다음과 같은 두 프로세스가 존재

- PID 0: 커널이 직접 만든 swapper
- PID 1: init이라는 사용자 프로세스의 부모. 고아 프로세스가 생길 때도 얘가 정리해줌

**왜 복사로 생성하나?** 시스템의 모든 환경 설정을 매번 새롭게 설정하는 게 아니라 부모로부터 물려받아 바로 시작

```c
pid_t pid = fork(); // 여기서 복제가 일어남

if (pid == 0) {
    // 나는 자식이다! (fork의 반환값이 0)
    // 여기서 exec()를 호출해 새 옷을 입음
} else {
    // 나는 부모다! (fork의 반환값이 자식의 PID)
    // 자식이 일을 마칠 때까지 기다리거나 (wait) 내 일을 함
}
```
부모가 fork()를 호출하면 복사가 일어나고, 부모와 자식 모두 반환값을 받는다. 이때 자식도 부모의 PC를 복사받아 fork() 다음의 명령어를 가리키고 있다.

## 프로세스 서술자와 태스크 구조체

```c
struct task_struct {
    #ifdef CONFIG_THREAD_INFO_IN_TASK
        /*
        * For reasons of header soup (see current_thread_info()), this
        * must be the first element of task_struct.
        */
        struct thread_info		thread_info; // CPU 관련 저수준 데이터들. 이전에는 스택 하위에 관리했지만 스택 오버플로로 중요 정보가 손상되어, 이제는 요기서 관리함
    #endif
        unsigned int			__state; // 프로세스의 상태

        /* saved state for "spinlock sleepers" */
        unsigned int			saved_state; // 스핀락(CPU를 점유) 대기 시간이 너무 오래 걸리면 잠시 sleep할 수 있게 만듦. 이때 이전 상태 저장을 위한 변수

        /*
        * This begins the randomizable portion of task_struct. Only
        * scheduling-critical items should be added above here.
        */
        randomized_struct_fields_start

        void				*stack; // 프로세스의 커널 스택 주소
        refcount_t			usage; // 구조체가 얼마만큼 참조되고 있는지. 0이 되면 자원을 반환
        /* Per task flags (PF_*), defined further below: */
        unsigned int			flags; // 프로세스의 특성을 저장
        unsigned int			ptrace;
    ...
}
```

### 핵심 필드

- 프로세스의 현재 상태와 정체성

`__state`: 현재 프로세스가 실행 중(TASK_RUNNING)인지, 자고 있는지 등을 나타냅니다.


`usage`: 이 구조체가 얼마나 참조되고 있는지 관리하며, 0이 되면 자원을 반환합니다.

`flags`: 프로세스의 특성(예: 커널 스레드인지 여부 등)을 비트 단위로 저장합니다.

- 실행 위치와 스케줄링 (가장 중요한 부분)

`*stack`: 이 프로세스의 커널 스택 주소입니다.

`prio, static_prio`: 스케줄러가 누구를 먼저 실행할지 결정할 때 쓰는 우선순위 값들입니다.


`se, rt, dl`: 실제 스케줄링 알고리즘(CFS, Real-time 등)이 사용하는 데이터들이 덩어리째(struct) 들어있습니다.

- CPU와 관계

`on_cpu`: 이 프로세스가 현재 어떤 CPU 번호에서 실행 중인지 나타냅니다.

`cpus_ptr, cpus_mask`: 이 프로세스가 실행될 수 있는 CPU들의 목록입니다(CPU Affinity).

### (p.74) 지금은 왜 thread_info도 스택 밑바닥/꼭대기가 아닌 task_struct 구조체 내부에 있는가?

- 스택 오버플로 방지
- 스택에 묶여있으면 스택 메모리를 할당할 때 반드시 연속된 물리 메모리 페이지가 필요
→ 분리 후에는 커널 스택을 가상 메모리 공간에 매핑할 때 물리 메모리 파편화 상관 없이 유연하게 할당하고 크기를 조절할 수 있게 됨
- [그 당시 업데이트를 정리한 아티클](https://lwn.net/Articles/692953/)



| 시대               | 커널 스택 하단 상태                | 특징 |
|--------------------|----------------------------------|------|
| 2.4 커널 이전      | `task_struct` 직접 위치           | 프로세스 생성 시 커널 스택과 PCB를 한 번에 할당 |
| 2.6 ~ 4.8 커널     | `thread_info` 위치               | 슬랩 할당자 도입으로 `task_struct`는 동적 할당, 스택 하단에는 요약 정보인 `thread_info`만 유지 |
| 최신 커널 (4.9~)   | 없음 (비어있음)                  | 보안 및 유연성을 위해 `thread_info`를 `task_struct` 내부로 이동 (`CONFIG_THREAD_INFO_IN_TASK`) |

> 슬랩 할당자: 자주 생성/삭제되는 객체를 빠르고 효율적으로 관리하기 위한 메모리 할당 방식. 자주 쓰는 구조체(task_struct, inode 등)를 미리 만들어서 캐시에 저장

## 프로세스 상태

### (p.77) 왜 TASK_RUNNING 플래그가 실행 중, 대기 중을 모두 포함하는가?

프로세스는 실행 중, 대기 중 모두 같은 플래그로 상태를 표시한다. 왜?

- 커널 입장에서는 둘 다 ‘스케줄링 대상’이라는 점이 동일하기 때문.
- 또한, 하드웨어적으로는 구분됨. ‘실행 중’ 프로세스는 current 매크로가 가리키고 있으며 ‘대기 중’ 프로세스는 실행 대기열(Runqueue)에 들어있음
- 두 상태의 전환은 컨텍스트 스위칭 시 일어난다. 컨텍스트 스위칭은 매우매우 빈번하게 일어나는데 그때마다 플래그를 업데이트하면 또다른 병목이 생길 것 (메모리 배리어, 캐시 라인 바운싱 → 캐시 무효화됨... 이런 과정까지 거치니)
