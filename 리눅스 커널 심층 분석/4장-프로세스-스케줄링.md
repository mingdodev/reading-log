# 4. 프로세스 스케줄링

## 멀티 태스킹

유닉스는 태초부터 선점형 멀티태스킹을 지원했으며, 리눅스도 선점형 멀티태스킹을 지원한다.

### 98p. 협동형 멀티태스킹 운영체제에서는, 그럼 유저 공간이 어떻게 동작하는 거지?

> MacOS 9 이전, 윈도우 3.1 이전

선점형의 경우 여러 프로세스를 같은 비중으로 실행할 수 있으니 유저가 여러 프로그램을 사용할 수 있는데, 협동형은 어떻게 되는 것인가? 하나의 거대한 프로세스가 양보하지 않으면 다른 프로그램은 먹통이 되는 거?

- 맞다. 다른 프로그램은 완전히 멈추고 마우스 커서도 안 움직인다.
- 그래서, 잘 만들어들은 프로그램이 **주기적으로 양보하도록 설계**함

```c
// 모든 Windows 3.1 앱의 핵심 루프
while (GetMessage(&msg, NULL, 0, 0)) {  // ← 여기서 자동으로 양보 발생
    TranslateMessage(&msg);
    DispatchMessage(&msg);              // ← 이벤트 처리
}

// MacOS는 WaitNextEvent() 함수
```

- 메시지 큐 처리, 시스템 콜 호출 시 OS에 제어권이 넘어가 다른 앱이 실행될 수 있다. 또는 명시적 yield 호출도 가능.

## 리눅스 프로세스 스케줄러 정책

입출력 프로세스에 관대하나 프로세서중심 프로세서를 소홀히 하지 않는다. 응답 시간을 빠르게 하는 것, 시스템 사용률을 최대화하는 것 사이 트레이드 오프를 어떻게 줄였나?

<br>

대부분의 운영체제들은 **우선순위**와 남아있는 **타임슬라이스** 값을 고려해 선점을 정한다. 우선순위는 값(나이스, 실시간)을 매겨 순서를 정하며, 동일 우선순위는 라운드 로빈으로 실행한다.

그러나 리눅스의 CFS 스케줄러는 **프로세스별로 타임슬라이스 값을 할당하는 대신 프로세서 할당 비율을 지정**한다. 프로세서 할당 비율은 나이스 값으로 결정된다.
리눅스는 각 프로세스가 CPU를 얼마나 사용했는지를 추적해, 가장 적게 실행된 프로세스를 다음으로 실행한다.
> 오 이거 서버 CPU 사용률 낮추려 했을 때 실제로 그랬었다

<br>

대화형 작업은 빠른 반응 속도가 보장되어야 하므로 필요한 순간에 프로세스 시간을 얻을 수 있어야 한다. 대부분의 운영체제들은 대화형 작업에 높은 우선순위(나이스 값)와 긴 타임슬라이스를 주어 이 목표를 달성하지만, 리눅스는 다르다. 나이스 값이 같더라도 대화형 작업의 응답성을 높일 수 있다.

CFS는 나이스 값이 같은 모든 프로세스에 동일한 CPU 시간을 보장한다. 대화형 작업은 실제로 시간을 거의 안 쓰기 때문에 항상 덜 쓴 상태가 된다. 따라서 입력이 들어오는 순간 높은 우선순위로 선점할 수 있어, 공평함을 유지하면서도 응답성을 챙긴다.

### p.103 CFS가 타임슬라이스 대신 vruntime을 쓰는 이유가 특별히 있나?

> 간단하고 똑똑하긴 한데, 단지 그것 때문인가?
당시 이유를 찾아봤을 때는 프로세스 하나하나에 타임슬라이스를 할당하는 방식이 복잡해서 그랬던 것 같긴한데..

타임슬라이스 기반 스케줄러는 구조적인 문제가 있다.

- 슬라이스가 짧으면 → 컨텍스트 스위칭 오버헤드 증가
- 슬라이스가 길면 → 응답성 저하
- 대화형 작업 감지를 위해 휴리스틱(경험적 보정값)이 계속 쌓임

CFS 설계자 Ingo Molnár는 이를 없애고 싶었다. 실제로 2007년 패치 공지에서
"CFS has no notion of timeslices and has no heuristics whatsoever"라고 밝혔다.

대신 CFS는 질문을 바꿨다.
"얼마나 줄까?" → "누가 가장 덜 받았나?"

vruntime(가상 실행 시간)을 기준으로 가장 적게 실행된 프로세스를 다음에 실행한다.
대화형 작업은 실제로 CPU를 거의 안 쓰므로 항상 vruntime이 낮은 상태가 되고,
입력이 들어오는 순간 자연스럽게 선점 우선권을 갖는다.

공평함 자체가 응답성의 원천이 되는 구조다.

> 참고: [Ingo Molnár의 CFS 패치 공지 (LKML, 2007)](https://lkml.indiana.edu/hypermail/linux/kernel/0703.2/0475.html)  
> 참고: [Linux 공식 CFS 문서](https://kernel.org/doc/html/v5.17/scheduler/sched-design-CFS.html)

## 리눅스 스케줄링 알고리즘

### p.104 리눅스 6.6 이후의 EEVDF 스케줄러

지금은 CFS가 EEVDF 스케줄러로 대체되었다. CFS는 뭐가 문제였나?

**레이턴시.** "얼마나 급한가"를 직접 표현할 방법이 없는 설계

<br>

리눅스는 선점형 커널이라 언제든 프로세스를 교체할 수 있을 것 같지만,
그 이전에 스케줄러가 판단하는 '선점 조건'이 따로 있다.
CFS의 선점 조건은 다음과 같다.

```
curr.vruntime - new.vruntime > sched_wakeup_granularity
```

<details>
<summary>v3.19 실제 코드</summary>

```c
s64 vdiff = curr->vruntime - se->vruntime;
// 현재 실행 중인 태스크 vruntime - 새로 깨어난 태스크 vruntime
// 양수면 새 태스크가 덜 쓴 상태

if (vdiff <= 0)
    return -1;  // 새 태스크가 오히려 더 많이 썼으면 → 선점 불가

gran = wakeup_gran(curr, se);  // 임계값 계산 (sched_wakeup_granularity 기반)

if (vdiff > gran)
    return 1;   // 차이가 임계값보다 크면 → 선점 가능

return 0;       // 차이가 임계값 이하 → 선점 안 함

---

wakeup_gran(struct sched_entity *curr, struct sched_entity *se)
{
	unsigned long gran = sysctl_sched_wakeup_granularity;

	/* 임계값 자체도 경험적 값이고,
   여기서 추가로 nice 가중치에 따라 gran을 보정하는 것도 휴리스틱.
   가벼운(우선순위 높은) 태스크일수록 선점이 쉬워진다. */
	return calc_delta_fair(gran, se);
}

---

static inline u64 calc_delta_fair(u64 delta, struct sched_entity *se)
{
	if (unlikely(se->load.weight != NICE_0_LOAD))
		delta = __calc_delta(delta, NICE_0_LOAD, &se->load);

	return delta;
}
```

</details>

<br>

단순히 CPU를 덜 썼다고 선점이 되는 것이 아니라, 임계값 만큼 차이가 나야 `need_resched` 플래그가 세팅되고 선점이 일어난다. **컨텍스트 스위칭 비용을 줄이기 위해** 충분히 불공평해졌을 때만 선점하는 것이다.

따라서 오디오 태스크와 같이 수ms 정도로 반응성이 필요한 태스크의 경우 아직 완전 불공평하진 않네~라고 판단해 버려 대기 -> 소리가 끊기는 치명적인 문제가 발생할 수 있는 것이다. 즉 **얼마나 급한가**를 표현할 수 있는 구조가 없었고, 이게 레이턴시 문제이다.

그동안은 이 문제를 해결하기 위해 여러 가지 휴리스틱한 패치들을 사용해왔다. 그러나, 패치가 쌓일수록 코드 복잡. 구조 자체를 개선하는 더 나은 방안이 있을 것 -> 2023년 Peter Zijlstra가 EEVDF 패치를 제안, 리눅스 6.6에 병합

<br>

- [스케줄러 첫 소개](https://lwn.net/Articles/925371/)
- [EEVDF 스케줄러 완성](https://lwn.net/Articles/969062/)

EEVDF는 각 태스크에 lag 값을 부여한다.
lag가 0 이상인(CPU 시간을 아직 다 못 받은) 태스크 중
**가상 데드라인**이 가장 이른 태스크를 먼저 실행한다.
태스크마다 타임슬라이스를 직접 지정할 수 있고,
짧은 타임슬라이스 = 이른 가상 데드라인 = 먼저 선점.
"얼마나 급한가"를 구조적으로 표현할 수 있게 됐다.

> 정말 쉽게 말하면 얼마나 급한가를 가상 데드라인이라는 개념 도입으로 표현할 수 있게 되었다고 이해하면 될듯

## 선점과 컨텍스트 전환

선점은 스케줄러 호출을 통해 일어날 수 있다. 리눅스는 완전 선점형 운영체제로, 사용자 공간으로 전환될 때뿐만 아니라 커널 공간에서도 스케줄러 호출이 가능하다. 즉, 선점이 좀 더 적극적으로 일어날 수 있다.

### 사용자 선점

커널이 사용자 공간으로 돌아가는 순간은 커널이 대기하게 되므로, **프로세스를 전환해도 상관 없는 상태**이다. 따라서 이때 `need_rescheduled` 플래그를 통해 선점이 필요한 프로세스가 있는지 확인 후, 스케줄러를 호출할 수 있다.

- 시스템 콜 이후
- 인터럽트 처리 끝

### 커널 선점

비선점형 커널에서는 커널 코드가 완료될 때까지 실행된다. 스케줄러가 개입할 수 없다. 그러나 리눅스 커널에서는 **스케줄링을 다시 해도 안전한 상태**라면 어떤 순간에도 선점이 가능하다.

- `preempt_count`가 0일 때

다중 진입이 가능한 영역을 '잠금'을 통해 표시한다. 이는 `thread_info`의 `preempt_count` 카운터 값을 통해 구현된다. 잠금이 걸릴 때마다 1씩 증가하고, 풀릴 때마다 1씩 감소한다. 0이면 `need_rescheduled` 플래그를 확인하고 설정됐으면 스케줄러를 호출한다.

또는 
- `scheduled` 함수 직접 호출
