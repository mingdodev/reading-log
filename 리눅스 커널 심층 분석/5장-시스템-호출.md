# 5. 시스템 호출 (System Call)

시스템 콜은 **레이어**로, 세 가지 역할을 함

1. 사용자 공간에 하드웨어를 추상화된 형태로 제공
2. 시스템 자원을 애플리케이션으로부터 보호
3. 이 레이어를 통해 자원을 가상화해서 쓸 수 있음

애플리케이션은 시스템 콜을 직접 사용하는 게 아니라 API를 통해 사용함
리눅스의 시스템 콜도 유닉스 기반 운영체제 간 공통 이식성을 위한 POSIX 표준을 따름. POSIX C 표준 라이브러리를 구현한 것이 `glibc`

### p.137 C의 특성 상 다른 프로그래밍 언어의 프로그램에서 사용할 수 있게 감싼 형태를 쉽게 만들 수 있다. C의 언어적 특성이 뭔데?

> 네이티브 모듈? 을 잘 제공한다는 뜻인 것 같은데 그래서 뭐 때문?

1. Name mangling 없음: 컴파일 후에도 함수 이름이 그대로
2. 런타임 없음: 순수 기계어로, 어떤 언어 런타임에 올려도 충돌 없음
3. Calling convention 단순: 컴파일 시 함수 호출 구조가 매우 단순
4. 타입 시스템 단순: C의 타입은 크기가 명확. 객체, 클래스, 제네렉 X

<details>
<summary> Calling convention 이란?</summary>

함수 호출 시 인자와 반환값을 CPU 레지스터와 스택에 어떻게 배치할지에 대한 규칙

#### C — 단순하고 예측 가능

```c
int add(int a, int b) { return a + b; }
```

컴파일 후 어셈블리:
```asm
add:
    ; 인자: rdi=a, rsi=b  (규칙이 고정)
    mov eax, edi
    add eax, esi
    ret              ; 반환값: rax에 저장 (규칙이 고정)
```

#### C++ — 숨겨진 인자가 생긴다

```cpp
class Foo {
    int add(int a, int b) { return a + b; }
};
```

메서드 호출 시 **this 포인터가 자동으로 첫 번째 인자**로 삽입됨. 외부 언어는 이걸 몰라서 연동이 어렵습니다. 그래서 C++ 라이브러리를 외부에 노출할 때 `extern "C"`로 C 방식으로 강제합니다.

#### Java — JVM 스택 기반, 완전히 다른 세계

CPU 레지스터가 아닌 **JVM 가상 스택** 사용. 외부 언어가 직접 호출 불가능 → JNI라는 별도 브릿지 필요

#### Python — 모든 것이 객체, 런타임이 개입

a, b가 전달될 때 단순한 int가 아니라 **PyObject*** (참조 카운터 + 타입 정보 + 값)로 감싸짐. 단순한 숫자 하나도 이 구조체로 감싸져서 전달되기 때문에 C와 직접 연결 불가능.

#### Go — 여러 반환값, 자체 스택 방식

반환값이 여러 개 가능 (C는 rax 하나뿐). Go 고루틴은 자체 관리하는 작은 스택 사용 → C와 직접 연결 시 스택 전환 비용 발생 (CGo가 느린 이유)

#### 한눈에 비교

| 언어 | 인자 전달 | 반환값 | 런타임 개입 | C와 직접 연결 |
|------|---------|--------|-----------|-------------|
| C | 레지스터, 규칙 고정 | rax 하나 | 없음 | ✅ 바로 가능 |
| C++ | C와 같지만 this 숨겨짐 | rax | 없음 | ⚠️ extern "C" 필요 |
| Java | JVM 가상 스택 | JVM 스택 | JVM 필수 | ❌ JNI 브릿지 필요 |
| Python | PyObject* 구조체 | PyObject* | CPython 필수 | ❌ CPython API 필요 |
| Go | 레지스터 | 여러 개 가능 | GC, 고루틴 스택 | ⚠️ CGo 필요, 느림 |

**결론**: C만 "그냥 기계어 함수"이기 때문에 다른 언어들이 C를 공용 인터페이스로 채택한 겁니다.

</details>

## 시스콜

### p.138 `asmlinkage`가 뭐임?

인자를 레지스터가 아닌 스택(커널 스택)에서 읽으라고 알려주는 매크로

왜 시스템 콜은 스택을 쓰죠?

일반 함수 호출은 컴파일러가 호출자, 피호출자 양쪽 코드를 다 생성한다.
즉, 양쪽이 같은 규칙을 공유하니 레지스터로 주고 받으면 됨.
같은 프로세스 내에서 함수 호출이 일어나니 레지스터에 인자를 넣어뒀다 연산하고... 반복 ...

그러나 시스템 콜은 호출자가 어셈블리: 유저스페이스에서 syscall을 날리는 건 `glibc` 내부의 어셈블리 코드이고, 피호출자는 커널 내부의 C 코드. 시스템 콜이 일어나면 CPU가 함수 호출 인자를 스택에 넣어준다. 그렇기에 커널 내부의 C 코드에게 그걸 알려서 인자를 전달받도록 하는 것.

이게 규칙인가? 그렇다.

**유저공간 어셈블리(glibc) ↔ 커널 C 함수 사이의 인자 전달 규칙**이다.

일반 함수 호출: 컴파일러가 호출자 + 피호출자 양쪽의 코드를 생성
→ 둘 다 같은 규칙을 알고 있음 → 레지스터로 주고받으면 됨

시스템 콜: 호출자는 glibc의 어셈블리 코드이다. 즉 컴파일러가 만든 코드가 아님
피호출자는 커널의 C 코드로, 둘 사이의 함수 호출 인자 전달은 스택에서 한다.
시스템 콜이 일어나면 CPU가 함수 호출 인자를 스택에 넣어준다.
→ asmlinkage 를 붙여서 해당 함수의 인자들은 스택에서 읽어야 함을 알려줌

140p에 슬쩍 언급이 되어있는데 (sysenter) 스택에 push하는 시점도 아키텍처별로 다르다고한다.. ([또 그 사이트](https://lwn.net/Articles/604515/))

- 32비트 int 0x80 (시스콜이 소프트웨어 인터럽트 명령어의 일부)
	```nasm
	; 유저공간에서 직접 스택에 push
	push 6          ; 인자3
	push buf        ; 인자2
	push 1          ; 인자1 fd
	mov eax, 4      ; syscall 번호 (write)
	int 0x80        ; 소프트웨어 인터럽트로 커널 진입
	```
	- 유저공간에서 직접 스택에 push했다고 함

- 64비트 현대 (시스콜은 별도 명령어로 분리)
    
    ```nasm
    ; 유저공간에서 레지스터에 넣고 syscall
    mov rax, 1      ; syscall 번호
    mov rdi, 1      ; 인자1 fd
    mov rsi, buf    ; 인자2
    mov rdx, 6      ; 인자3
    syscall         ; 전용 명령어로 커널 진입
    ```
    
    - syscall 명령어가 따로 있고, 커널 진입점(어셈블리 코드)에서 레지스터 값들을 커널 스택에 저장
    - int 0x80보다 빠름

## 시스템 호출 핸들러

### 시스템 아키텍처별로 syscall 인자를 저장할 수 있는 레지스터의 개수가 다름

지금은 6개까지 저장 가능, 초과하면 구조체를 만들어서 저장 후 레지스터 하나에 구조체의 주소를 저장

## 시스템 호출 구현

- 목적 정의, 인자와 반환값과 오류 코드 정의 / 책임의 분리, 이식성, 하위 호환성 등 기본적인 설계 원칙이 일반적인 API 설계와 유사하지만서도 깊이가 다른 느낌
- 훨씬 더 멀리 봐야 하고, 하위 호환성과 이식성에 대해 깊이 있게 생각해야 한다.
- 커널 공간에서 사용하기 때문에 매개변수 검사가 더 꼼꼼히 이뤄져야 한다. (시스템 보안, 안정성)

### 시스콜 구현 구조

```c
ssize_t ksys_write(unsigned int fd, const char __user *buf, size_t count) // __user는 이 포인터가 유저공간 메모리를 가리킨다는 표시 (커널이 직접 역참조하면 안 된다는 경고)
{
	CLASS(fd_pos, f)(fd); // CLASS: 자동으로 파일 참조 카운터를 관리하는 매크로. 파일 디스크립터 번호로 파일 객체를 가져옴
	ssize_t ret = -EBADF; // 반환 기본 값을 에러코드로 설정해두고, 성공하면 덮어쓰는 구조

	if (!fd_empty(f)) {
		loff_t pos, *ppos = file_ppos(fd_file(f));
		if (ppos) {
			pos = *ppos;
			ppos = &pos;
		}
		ret = vfs_write(fd_file(f), buf, count, ppos);
		if (ret >= 0 && ppos)
			fd_file(f)->f_pos = pos;
	}

	return ret;
}

// 매크로: asmlinkage 붙여주기, 보안 검사 코드 자동 삽입, syscall 테이블에 등록. sys_write() 생성
// 유저공간 진입점 전용 매크로
SYSCALL_DEFINE3(write, unsigned int, fd, const char __user *, buf,
		size_t, count)
{
	return ksys_write(fd, buf, count);
}
```
매크로와 그냥 함수로 분리한 이유?
- 커널 내부에서 write 기능이 필요할 때는 ksys_write()를 바로 호출할 수 있음. sys_write()는 시스템콜을 위한 것으로 부가 기능(데코레이터 패턴+어댑터 패턴과 유사한 느낌)이 붙어있으며, `asmlinkage` 때문에 호출 방식도 달라서 문제가 될 수 있다. (커널 내부끼리는 레지스터로 전달받는데 스택을 읽으면 엉뚱한 곳에서 읽게 됨)
- 저 매크로는 인자 개수를 컴파일 타임에 알아야 하기 때문에 0~6까지 인자 개수별로 미리 정의해뒀다고 함

## 시스템 호출 컨텍스트

### p.147 syscall 실행하는 동안 커널은 프로세스 컨텍스트 상태에 있으며, 완전히 선점 가능하다.

앞서 시스템 호출 번호, 시스템 호출 시의 매개변수는 레지스터에 저장한다고 했다.

```
mov rax, 1 # 레지스터에 상태 값 저장함 (유저 모드)
syscall # 시스템 호출 (커널로 진입)
시스템 콜 진행 중... # 여기서도 선점이 가능하다는 뜻
```
그렇다면 시스템 콜 실행할 때도 레지스터를 따로 저장하고 복원가능해야겠네? -> task_struct에서 수행한다고 함.
그럼 여기서도 경쟁이 발생할 수 있겠네? 동일한 자원에 여러 개의 시스템 콜이 접근하려면?
맞음. 커널도 동기화 기법(뮤텍스/스핀락/세마포어 등) 필요함.
애초에 선점형이 아니더라도 멀티 코어인 경우 무조건 필요.

이런 구조의 장점: 선점 가능성으로 인해 응답성이 좋아질 수 있음
단점: 경쟁에 대한 해결책 필요

커널은 여기서 잘못되면 진짜 답이 없음.
그래서 요즘 Rust로 커널 드라이버를 작성하려는 게 엄격한 타입/소유권 시스템으로 컴파일 시점에 문제를 찾으려고.. 근데 데드락 같은 건 언어가 잡을 수 있는 게 아니니 설계가 정말 중요하다.

### p.148 시스콜이 모듈화되면 안 되는 이유?

시스콜은 모듈이 아닌 커널 이미지로 컴파일된다.

1. 시스콜은 필요할 때 로드되는 게 아니라 항상 존재를 보장해야 함
> 아키텍처에 따라 다르거나 상황에 따라 존재하지 않으면 치명적인 문제가 되지 않나

2. 닭이 먼저냐 달걀이 먼저냐: 모듈 로드 자체가 syscall을 사용함
```
insmod 실행
  → open() syscall   ← .ko 파일 열기
  → read() syscall   ← 파일 읽기
  → 모듈 등록 syscall

그런데 syscall이 모듈이면?
  → syscall 로드하려면 syscall이 필요 💥
  → 부팅 자체가 불가능
```
하지만 부팅에 필요하지 않은 것들은 유연하게 모듈화할 수도 있지 않나? (하이브리드 커널의 원리처럼)

3. 런타임에 로드된다면, 악성 모듈에 의해 시스콜 테이블을 가짜로 교체할 수도 있다.

해킹!! 실제로 루트킷이 syscall 테이블을 후킹하여 동작한다고 함

그 외에도 굳이 모듈화해서 얻는 장점이 없다. 시스콜 테이블을 일부만 떼어내는 것도 어렵고, 전부 다 합쳐도 크기가 작아서 굳이 복잡성을 더할 필요가 없음.

> 애플리케이션 개발 시에는 모듈화 및 필요할 때만 필요한 걸 쓰자는 철학이 대부분의 문제에 좋은 해결책이 되는 경우가 많은 듯. 그렇지만 커널과 같이 잘못되었을 때 정말 치명적이고, 보안이 중요한 상황에서는 굳이 유연한 구조가 필요 없으며, 단순하고 명확한 구조로 가야한다. 다른 특수한 문제 상황에서도 적용될 듯. 뭐든 문제의 본질을 알고 설계해야 함

### 자바에서 시스템 콜 호출은 JVM을 통해 네이티브 c를 호출

진짜 저수준이 필요하면 JNI 또는 Panama

**Panama (Java 22+)**
```java
// JNI 없이 C 함수 직접 호출 가능
MethodHandle write = Linker.nativeLinker()
    .downcallHandle(/* write 함수 */);
```
- JNI의 엄청난 보일러플레이트 (자바로 클래스 선언, C 헤더 생성, 구현 파일 작성 등) 없음
- C 코드를 작성하고 컴파일, .so 빌드하지 않아도 됨

자바 생태계에 없는 고성능 C 라이브러리가 필요할 때 쓴다 (머신러닝, 수치 연산, OpenSSL 보안 등)

```
게이밍:

  scx_layered → 워크로드 종류별 레이어 분리

오디오/멀티미디어:
  레이턴시 최소화 전용 스케줄러
```
용도에 따라 커스텀해서 최적화해서 쓰이고 있는 듯하다
